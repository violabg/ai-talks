## 1. Il problema fondamentale

L'AI è come un nuovo assunto brillante ma **senza memoria**: ogni sessione riparte da zero. Non sa niente del tuo progetto, delle tue convenzioni, delle decisioni passate. Se non gli dai il contesto giusto, producerà codice generico, inconsistente o addirittura in conflitto con quello che hai già scritto.

Questo non è un limite temporaneo — è strutturale. I modelli LLM non hanno stato persistente tra sessioni. La soluzione non è aspettare modelli "più intelligenti": è **progettare il tuo workflow e la tua codebase in modo da rendere il contesto accessibile e comprensibile all'AI**.

> Il problema non è l'AI. È che lavoriamo con lei come se avesse memoria, quando invece dobbiamo lavorare come se ogni volta fosse il primo giorno.

## 2. I 5 errori comuni

### Errore 1 — Affidarsi all'output senza verificarlo

L'AI genera codice plausibile, non codice corretto. La plausibilità è il suo punto di forza e il suo punto di debolezza. Impara a leggere criticamente ogni output prima di integrarlo.

**Soluzione**: Tratta l'AI come un junior developer: rivedi sempre il codice, non fare merge ciechi.

### Errore 2 — Non fornire contesto sufficiente

"Scrivi una funzione che fa X" produce risultati mediocri. L'AI non conosce il tuo stack, le tue librerie, le tue convenzioni di naming, i tuoi vincoli architetturali.

**Soluzione**: Prima di ogni prompt significativo, condividi: struttura della codebase rilevante, interfacce esistenti, convenzioni da rispettare, vincoli tecnici.

### Errore 3 — Sessioni troppo lunghe senza reset del contesto

Più una sessione si allunga, più il modello "dimentica" le istruzioni iniziali e inizia ad accumulare contraddizioni. Il contesto si degrada.

**Soluzione**: Usa sessioni brevi e focalizzate. Al cambio di task, apri una nuova sessione e ricostruisci il contesto rilevante.

### Errore 4 — Mancanza di architettura agent-friendly

Una codebase con file da 2000 righe, naming ambiguo, dipendenze circolari e zero documentazione è incomprensibile per l'AI — e spesso anche per te dopo 6 mesi.

**Soluzione**: Struttura la codebase per la navigabilità. Vedi la sezione 3 per i dettagli.

### Errore 5 — Non documentare le decisioni di progetto

L'AI non sa perché hai scelto questa libreria, perché hai scartato quell'approccio, perché il modulo X è strutturato in quel modo. Senza questa memoria contestuale, rischi di far riproporre soluzioni già scartate.

**Soluzione**: Mantieni un `DECISIONS.md` o ADR (Architecture Decision Records) nel repo. L'AI può leggerli se glieli fornisci.

## 3. Architettura agent-friendly della codebase

Una codebase leggibile dall'AI condivide le stesse caratteristiche di una codebase leggibile dagli umani — ma alcune proprietà diventano ancora più critiche.

### Struttura modulare

Ogni modulo ha una responsabilità chiara, esposta attraverso un'interfaccia esplicita. L'AI capisce cosa fa un modulo dal nome e dall'interfaccia pubblica, senza dover leggere l'implementazione.

![Struttura modulare della codebase](/images/articles/arch-1-modular.png)

Regola pratica: se non riesci a spiegare in una riga cosa fa un modulo, probabilmente fa troppe cose.

### Convenzioni di naming coerenti

Nomi ambigui costringono l'AI a fare inferenze — e le inferenze sono spesso sbagliate. Usa naming esplicito, anche se più lungo.

![Convenzioni di naming](/images/articles/arch-2-naming.png)

Esempi:
- `getUserData()` → ambiguo. `fetchUserProfileFromDB()` → chiaro.
- `handleClick()` → ambiguo. `handleSubmitLoginForm()` → chiaro.
- `utils.ts` con 400 righe → problema. `formatDate.ts`, `validateEmail.ts` → meglio.

### Single Responsibility nei file

File corti e focalizzati sono più facili da includere nel contesto di un prompt. Un file da 100 righe con una sola responsabilità è molto più utile di uno da 800 con funzioni miste.

![Single responsibility nei file](/images/articles/arch-3-single-responsibility.png)

Soglia pratica: se un file supera le 200-300 righe, valuta se può essere diviso per responsabilità.

### README e documentazione inline di qualità

L'AI legge i commenti e i README. Un buon README a livello di modulo — anche breve — può fare la differenza tra output generico e output contestualizzato.

![Struttura dei README](/images/articles/arch-4-readme.png)

Cosa includere in un README di modulo:
- Scopo del modulo in 1-2 righe
- Dipendenze principali
- Decisioni architetturali rilevanti
- Esempi di utilizzo

## 4. Le 7 fasi del workflow

Un workflow strutturato riduce il rumore e massimizza la qualità dell'output AI. Ecco le 7 fasi.

| Fase | Azione | Perché |
|------|--------|--------|
| 1 | **Definire l'obiettivo con precisione** | Output ambigui partono da input ambigui |
| 2 | **Raccogliere il contesto rilevante** | File, interfacce, vincoli da includere nel prompt |
| 3 | **Scrivere un prompt strutturato** | Obiettivo + contesto + formato atteso + vincoli |
| 4 | **Validare l'output prima di integrarlo** | L'AI sbaglia: meglio scoprirlo prima del merge |
| 5 | **Iterare con correzioni mirate** | Prompt di correzione specifici, non generici |
| 6 | **Documentare la soluzione adottata** | Aggiorna README, commenti, ADR se necessario |
| 7 | **Aggiornare o scrivere i test** | I test sono la rete di sicurezza del ciclo successivo |

### Fase 1 — Definire l'obiettivo

Prima di aprire il prompt, chiarisciti l'obiettivo. Scrivi in una frase cosa vuoi ottenere. Se non riesci, il prompt sarà nebuloso.

### Fase 2 — Raccogliere il contesto

Seleziona i file rilevanti. Non tutto: solo quello che l'AI deve conoscere per rispondere bene. Un contesto rumoroso è peggio di un contesto scarso.

### Fase 3 — Prompt strutturato

Un buon prompt include:
- **Ruolo**: "Sei un senior developer TypeScript"
- **Obiettivo**: cosa deve fare
- **Contesto**: file rilevanti, interfacce, vincoli
- **Formato**: come deve essere la risposta

### Fase 4 — Validazione dell'output

Leggi il codice generato. Verifica che rispetti le convenzioni. Testa localmente prima di integrare. L'AI è veloce, la review deve esserlo altrettanto — ma non può essere zero.

### Fase 5 — Iterazione mirata

Se l'output non è corretto, non rigenerare da zero. Correggi con prompt specifici: "Il tipo di ritorno della funzione X dovrebbe essere Y, non Z." Le correzioni precise producono iterazioni migliori.

### Fase 6 — Documentazione

Se hai preso una decisione architetturale durante questa sessione, documentala. Non farlo significa che il prossimo developer — umano o AI — dovrà ricostruire il ragionamento da capo.

### Fase 7 — Test

Scrivi o aggiorna i test. I test sono il ciclo di feedback più affidabile. Sono anche contesto eseguibile che l'AI può leggere nelle sessioni successive.

## 5. Le 5 skill pratiche

### Skill 1 — Prompt Engineering

Scrivere prompt chiari è una competenza, non un'intuizione. Impara la struttura base: ruolo + obiettivo + contesto + vincoli + formato.

```bash
# Strumenti utili
# Cursor, GitHub Copilot, Claude, ChatGPT
# Tutti supportano system prompts configurabili
```

### Skill 2 — Context Management

Saper scegliere cosa includere nel contesto è la skill più sottovalutata. Troppo contesto rallenta e confonde. Troppo poco produce output fuori target.

Regola pratica: includi solo ciò che è direttamente rilevante per il task corrente. Aggiungi contesto per strati, non tutto in una volta.

### Skill 3 — Output Validation

Riconoscere gli errori dell'AI richiede conoscenza tecnica solida. L'AI è convincente anche quando sbaglia. Coltiva la tua capacità di review critica — non delegarla all'AI stessa.

### Skill 4 — Iterative Refinement

Le prime risposte raramente sono le migliori. Impara a fare iterazioni: correzioni specifiche, richieste di spiegazione, alternative. Il dialogo con l'AI è più efficace del prompt one-shot.

### Skill 5 — Documentation-First

Scrivi per l'AI, non solo per gli umani. Commenti chiari, README aggiornati, interfacce ben documentate — sono contesto che puoi fornire all'AI in ogni sessione futura.

```bash
# oppure usa le estensioni IDE per Cursor/Copilot
```

## Conclusione

Sviluppare con l'AI non significa smettere di programmare. Significa aggiungere una skill: quella di collaborare efficacemente con un assistente potente ma senza memoria.

Le basi sono semplici:
1. Dai contesto di qualità
2. Verifica sempre l'output
3. Struttura la codebase per la navigabilità
4. Lavora per cicli brevi con feedback espliciti
5. Documenta le decisioni

Il resto è pratica.
